# pi-orchestra v4 Bench — Phase 1 terminal spike verdict

**Date:** 2026-07-11  
**Branch:** `v4-bench`  
**Verdict:** **GO — embedded PTYs. Do not pivot to companion mode.**

The vertical slice hosts real Claude Code and Hermes TUI processes in daemon-owned
PTYs, captures bounded terminal state, serves it over a versioned Unix-socket
protocol, and renders both harnesses as styled cards in the ratatui STAGE. Closing
and reopening clients does not restart either harness. The client has an event-driven
idle path, synchronized-output frames, a tachyonfx-driven pulse on a braille cubic
Bézier, two themes, and a 72×30 stacked layout.

## What was built

- `orc-proto`: additive, versioned JSON messages for hello, snapshot, blocking
  output wait, input, resize, and latency ping. Empty/default cells use compact
  serde fields so replay is bounded enough for the input path.
- `orc-pty`: `portable-pty` host, 2,000-row bounded scrollback, 64 KiB read chunks,
  vt state, styled cell snapshots, input, resize, child lifecycle, and a shared
  condition-variable output signal.
- `orc-daemon` / `orcd`: multi-client Unix-socket server, daemon-owned PTYs,
  blocking change notification, replay after detach, bounded 1 MiB requests, and
  `tracing` instrumentation.
- `orc-app` / `pi-orchestra`: STAGE renderer, ember/phosphor tokens, wide and
  stacked layouts, card-sized PTY resize, braille-Bézier pulse, synchronized
  output, event-driven input/output threads, leader key, bracketed paste, SGR
  mouse forwarding, and a latency benchmark command.
- `rust/spikes/vt-bakeoff`: reproducible parser comparison rather than a prose-only
  dependency choice.

## Real harness fidelity

Tested live:

- Claude Code `2.1.198`, command `claude`.
- Hermes Agent `0.18.0`, command `hermes --tui`.

Both alternate-screen interfaces retained their recognizable layout, Unicode,
256/true color, cursor/input regions, and narrow reflow. The first capture exposed
an initial-size defect (daemon panes stayed at 90 columns and clipped inside the
cards); the client now sends each card's content size immediately and on resize.
The corrected captures were inspected frame-by-frame:

- `docs/v4-phase1-stage.gif` — real Claude + Hermes, wide ember.
- `docs/v4-phase1-narrow-phosphor.gif` — the same live panes at `stty cols 72
  rows 30`, stacked phosphor.
- `docs/v4-phase1-flood.gif` — four-pane output-flood capture.

The TestBackend suite renders the STAGE in both themes at 150×44 and exactly
72×30. Chrome is outside pane cells; terminal-native semantic colors remain inside
the pane rather than being recolored as theme chrome.

## VT parser bake-off

Versions: `vt100 0.16.2`, `termwiz 0.23.3`, `alacritty_terminal 0.26.0`.
The release benchmark processed 400 copies of a 120-frame corpus containing
alternate-screen switches, cursor addressing, true color, bold, CJK, and braille.

| Parser | Throughput | Replayable screen | Verdict |
|---|---:|---|---|
| vt100 | 165.0 MiB/s | yes | **selected** |
| termwiz | 233.1 MiB/s | no (escape actions only) | reject |
| alacritty_terminal | 238.5 MiB/s | yes | fallback |

`vt100` is slower, but 165 MiB/s is well beyond realistic harness output and its
small direct screen/cell API is already proven by both real harnesses. Termwiz does
not provide the required replayable terminal state by itself. Alacritty is faster
and more complete but pulls a substantially larger terminal/event surface into the
daemon. Keep it as the named fallback if the remaining raw-input/terminal suite
finds sequences vt100 cannot represent.

## Measurements

All binaries were release builds on this M-series Mac. Numbers are observations,
not targets inferred from implementation.

### Input and IPC latency

- Unix-socket ping, 5,000 samples: p50 **15 µs**, p95 **34 µs**, p99 **48 µs**,
  maximum **166 µs**.
- PTY input to visible replay, 100 samples against a daemon-hosted `cat`: p50
  **6.149 ms**, p95 **6.306 ms**, p99 **6.676 ms**, maximum **6.750 ms**.

The first version failed this gate: a blank 30×90 snapshot serialized every field
of every cell (356,840 bytes), producing roughly 540 ms visible latency. Compact
default-cell serialization reduced the same snapshot to 8,217 bytes and brought
the full measured path under the 16 ms budget.

### Idle

Five one-second `ps` samples with real Claude + Hermes and an attached idle client:

- `orcd`: **0.0% CPU** in every sample.
- `pi-orchestra`: **0.0% CPU** in every sample.
- Observed RSS: daemon about 2.9 MiB excluding child harnesses; client about
  7.7 MiB. Harness memory remains owned/accounted as child processes.

The client blocks on either terminal input or daemon output. There is no idle frame
poll. Animation uses 16 ms ticks only while the tachyonfx timer is active.

### Four-pane output flood

Four daemon PTYs ran unbounded `yes` producers while the client attached and
rendered. Five samples after the compact wire fix:

- `orcd`: **56.2–75.5% CPU**, about 66.7% mean; RSS **39.5–40.1 MiB**.
- `pi-orchestra`: **4.8–7.2% CPU**, about 5.7% mean; RSS **5.4–6.0 MiB**.

Memory remained bounded during the sample. This is a deliberately pathological
producer rate; Phase 2 still needs an hours-long soak and explicit output
coalescing/backpressure instrumentation before production sign-off.

### Detach and reattach

On the final real run, daemon PID `72162` retained child PIDs `72163` (Claude) and
`72164` (Hermes) across two independent client connections. Replayed sequences were
`(pane-1, 48)` and `(pane-2, 290/291)`; the Hermes sequence advanced independently
while both process identities remained unchanged. A daemon integration test repeats
the same attach/snapshot/drop/reattach flow with a fixture PTY.

## Terminal and input checks

- VHS/its PTY backend produced the inspected captures above.
- Ghostty live client: `TERM=xterm-ghostty`, active socket attachment.
- kitty `0.47.4` live client: `TERM=xterm-kitty`, active socket attachment.
  Kitty was installed specifically for the second named-terminal gate and is signed
  by a notarized Developer ID.
- Alacritty `0.17.0` was attempted first, but Homebrew warned that its cask fails
  Gatekeeper and `spctl` rejected it. No security bypass was attempted; the cask was
  removed and kitty used instead.
- macOS denied screen-capture creation for the named GUI terminal processes. Their
  live process environments and Unix-socket connections are recorded, but visual
  evidence for those exact apps is not claimed; visual evidence comes from VHS.

Input behavior in the spike:

- Kitty keyboard enhancement flags are negotiated through crossterm.
- Bracketed paste is forwarded with the standard begin/end markers.
- Mouse events inside pane cells are translated to content-relative SGR mouse bytes.
- `ctrl-g` is the only leader; a double tap sends literal `0x07`.
- Common keys are re-encoded after crossterm decoding. This is **not yet proof of
  byte-for-byte raw passthrough for every kitty extended key**. Phase 2 must move
  pane input to a raw-byte path (with a minimal leader recognizer) before claiming
  the prime directive is fully closed.

## Hermes adapter observations for Phase 5

`hermes --help` exposes:

- `--tui` and `--cli` interactive surfaces.
- `-z/--oneshot PROMPT`, documented as final response text only, with no banner,
  spinner, tool preview, or session-id line.
- `--resume SESSION` and `--continue [SESSION_NAME]`.
- provider/model/toolset overrides and a native `--worktree` option.

The help output does **not** advertise structured JSON events or exact usage/cost
records for `--oneshot`. Therefore Hermes is a valid headless text adapter but must
advertise `exact_usage: false` unless a later direct probe finds a documented usage
surface. Interactive Hermes remains a first-class STAGE pane regardless.

## Go/no-go decision and Phase 2 requirements

**GO with embedded PTYs.** Full-screen fidelity did not trigger the companion-mode
fallback. The architecture proved the existential pieces: daemon ownership,
multi-client attachment, replay, low idle CPU, sub-16 ms visible input, real harness
rendering, narrow reflow, themes, synchronized frames, and event-driven animation.

Phase 2 must not paper over the remaining spike limits:

1. Replace decoded/re-encoded keys with tested raw passthrough plus leader detection.
2. Add pid records, restart reaping, socket permissions/auth checks, and structured
   daemon log files.
3. Add output coalescing/backpressure metrics and the required hours-long soak.
4. Snapshot protocol tests must pin compact/default-cell compatibility.
5. Preserve vt100 as the default but keep the alacritty-terminal fallback decision
   explicit if real terminal-sequence fixtures fail.
6. Add reproducible Ghostty/kitty screenshots when macOS Screen Recording
   permission is available; do not reinterpret process evidence as image evidence.
